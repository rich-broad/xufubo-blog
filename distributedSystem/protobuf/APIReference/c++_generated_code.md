# C++ Generated Code
本文主要学习protobuf编译器生成的C++代码。  

## 编译器调用
举例如下：  
```C++
protoc --proto_path=src --cpp_out=build/gen src/foo.proto src/bar/baz.proto
```
这时将输出四个文件：build/gen/foo.pb.h, build/gen/foo.pb.cc, build/gen/bar/baz.pb.h, build/gen/bar/baz.pb.cc。编译器会自动创建bar，但是build/gen必须存在。  

## 包（Packages）
如果一个.proto文件包含包声明，则该文件的全部内容将被放置在相应的C++名称空间中。例如：  
```
package foo.bar;
```
文件中的所有声明都将放在foo :: bar命名空间中。  

## Messages
```
message Foo {}
```
将生成派生自[google::protobuf::Message](https://developers.google.com/protocol-buffers/docs/reference/cpp/google.protobuf.message#Message)的类。该类是个具体类，不存在未被实现的虚函数。Foo可能覆盖Message的非纯虚函数，也可能不覆盖，这取决于优化模式，默认情况下，Foo实现所有方法的专用版本以获得最大的性能。但是，如果.proto文件包含如下行：  
```
option optimize_for = CODE_SIZE;
```
那么Foo将只覆盖必要的最小功能集合，其他的依赖于基于反射的实现。这显着减少了生成的代码的大小，但也降低了性能。或者，如果.proto文件包含如下行：  
```
option optimize_for = LITE_RUNTIME;
```
这时，Foo将实现[google::protobuf::MessageLite](https://developers.google.com/protocol-buffers/docs/reference/cpp/google.protobuf.message_lite#MessageLite)接口。它只包含Message方法的一个子集。此时，只需要链接libprotobuf-lite.so库，而不是libprotobuf.so。lite库比full库小很多，更适合资源受限的系统，如手机。  
你不要去继承Foo类。因为为了提高性能，protobuf会做很多去虚拟化的事情，这会导致你覆盖的Foo的方法不起作用。这里不在详细描述。  
一条消息可以在另一条消息中声明。例如：message Foo {message Bar {}}。在这种情况下，编译器会生成两个类：Foo和Foo_Bar。另外，编译器在Foo内部生成一个typedef，如下所示：  
```
typedef Foo_Bar Bar;
```

## Fields
对于消息中的字段，protobuf会为其生成一系列的访问方法。除了一系列方法外，编译器还会为每个字段生成一个包含其字段号的整数常量。  
更加详细的信息参考官方的手册：[C++ Generated Code](https://developers.google.com/protocol-buffers/docs/reference/cpp-generated)。  

## Any
对于Any字段，如：  
```
import "google/protobuf/any.proto";

message ErrorStatus {
  string message = 1;
  google.protobuf.Any details = 2;
}
```
在生成的代码中，details字段的getter返回一个google::protobuf::Any的实例。  

## Oneof
```
oneof oneof_name {
    int32 foo_int = 4;
    string foo_string = 9;
    ...
}
```
如何生成，可以看看具体的代码。

```
enum Foo {

```

## Arena Allocation
Arena Allocation是一个C++专用功能，可帮助你在使用protobuf时优化内存使用并提高性能。  

## Services
为了使protobuf生成RPC调用需要的接口，需要在proto文件中指定如下行：  
```
option cc_generic_services = true;
```
protobuf只会生成接口，并不会绑定到具体的RPC实现上，需要你自己实现，我们接下来会分析brpc的实现。  
例如：  
```
service Foo {
  rpc Bar(FooRequest) returns(FooResponse);
}
```
这时protobuf编译器将生成Foo类表示服务，并为服务定义中定义的每个方法提供一个虚方法，对于Bar而言是：  
```C++
virtual void Bar(RpcController* controller, const FooRequest* request, FooResponse* response, Closure* done);
```
同时，也会生成一个stub，对于如上定义，将生成一个Foo_Stub类，该类是Foo的子类。客户端可以使用该类对远程的服务发起RPC调用。当然了，protobuf也只是生成了抽象的方法，并不会绑定任何具体的RPC实现。具体的实现，差异可能会很大。
